How I Built a Scalable REST API in 48 Hours Using Rust

personal2026-04-01

Last month I had a weekend hackathon challenge: build a production-ready REST API that could handle 50,000 requests per second with sub-10ms p99 latency. I chose Rust and the Axum framework, and what followed was one of the most intense and rewarding coding sessions of my career.

Why Rust for an API?

I'd been writing Go APIs for years, but a recent profiling session showed that a significant chunk of our latency came from garbage collection pauses at high throughput. Rust's ownership model eliminates GC entirely. Paired with Tokio's async runtime and Axum's ergonomic routing layer, I expected near-zero overhead above raw syscall costs.

The Architecture

The design was intentionally simple: an Axum router with middleware for authentication, rate limiting, and request tracing. State was managed via Arc<AppState> shared across handlers. For persistence, I used a connection-pooled interface over PostgreSQL via sqlx with compile-time query verification. Redis handled session tokens and rate limit counters.

Performance Results

After two days of development and tuning, the service achieved 62,000 requests per second on a single 8-core VM with p99 latency of 7ms under sustained load. Memory usage stayed flat at 48MB throughout the test. By contrast, a comparable Go implementation I had from a previous project peaked at 41,000 RPS with 14ms p99 and exhibited occasional 40–80ms spikes during GC pauses.

Lessons Learned

The Rust learning curve is real, but the compiler is an exceptional teacher. Every fight with the borrow checker forced me to think more carefully about data ownership and concurrency. I plan to migrate our highest-traffic internal service to this stack in Q3.
